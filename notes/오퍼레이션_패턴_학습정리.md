## 들어가며 — 4개 패턴의 큰 그림

이 4개는 모두 GoF 분류상 **행위 패턴(Behavioral Patterns)** 이다. 즉 "객체들이 *무엇을 하는가*(동작)"를 다룬다. 한 줄로 구분하면:

| 패턴 | 한 줄 정의 | 핵심 메커니즘 |
|------|-----------|--------------|
| **Command** | 요청(메서드 호출)을 객체로 캡슐화 | 동사를 명사로 |
| **Decorator** | 객체를 같은 타입으로 감싸 기능을 동적으로 추가 | 마트료시카(중첩) |
| **Iterator** | 순회 책임을 컬렉션에서 분리 | 내부 구조 은폐 |
| **Visitor** | 연산을 데이터 구조에서 분리 | 이중 디스패치 |

> **관통하는 통찰**: 행위 패턴은 본질적으로 "동작 = 함수"를 다루기 때문에, 함수가 1급 객체인
> 언어(또는 람다·generator·패턴매칭이 도입된 언어)에서는 패턴의 상당 부분이 언어 기능으로
> **흡수**된다. 이것이 구조 패턴(Composite, Bridge)·생성 패턴(Singleton, Factory)과의 결정적 차이다.

# 패턴 1: Command 패턴

## 1. 개념 — 왜 존재하는가

**한 줄 정의**: "요청(request)"을 객체로 캡슐화해서, 요청을 보내는 쪽과 받는 쪽을 분리한다.

### 문제 상황 (Lab12 리모컨)

```java
if (buttonName.equals("Light")) {
    // 전등 켜는 로직
} else if (buttonName.equals("Fan")) {
    // 선풍기 켜는 로직
}
```

진짜 문제는 "if-else가 많다"가 아니라 더 깊은 곳에 있다:

1. 요청을 다루는 주체와 요청 자체가 한 덩어리로 묶여 있다.
2. 요청을 **1급 시민으로 다룰 수 없다** — 변수에 담거나, 큐에 넣거나, 로그에 기록하거나, 되돌릴 수 없다.
3. **시간 분리(temporal decoupling)** 가 불가능 — "지금 만들고 나중에 실행"이 안 된다.

### 핵심 아이디어

> **메서드 호출을 객체로 바꾼다. (동사를 명사로)**

```java
Command cmd = new TurnOnLightCommand(light);  // 지금 안 실행
cmd.execute();                                 // 나중에 실행 가능
```

명령이 객체가 되면 저장·전달·큐잉·되돌리기·로깅·직렬화가 모두 가능해진다.

## 2. 구조 — 5가지 역할

| 역할 | 책임 | Lab12 예시 |
|------|------|-----------|
| **Command** (인터페이스) | `execute()` 선언 | `ICommand` |
| **ConcreteCommand** | 특정 Receiver의 action을 호출하도록 캡슐화 (얇은 래퍼) | `TurnOnLightCommand` |
| **Receiver** | 실제 작업 수행 | `Light`, `Fan` |
| **Invoker** | Command를 보관하고 실행을 관리 | `HomeAutomationRemote` |
| **Client** | ConcreteCommand 생성 + Receiver 지정 | `Demo` (main) |

**핵심**: Invoker는 Receiver를 **모른다**. Command(인터페이스)를 통해서만 통신한다.

### 역할별 "무지(無知)" 관계 — 이 패턴 이해의 핵심

```
Client    : 전부 안다. 무엇을(Command) 누가(Receiver) 시킬지 다 설계함.
Invoker   : 거의 모른다. "ICommand 하나"만 안다. execute()만 부른다.
            → 모르기 때문에, 어떤 명령이든 똑같이 다룰 수 있다 (보편성)
Receiver  : 자기 일만 안다. 자기가 Command 안에 들어있는지조차 모른다.
```

### "요청하는 자"는 Client인가 Invoker인가? — 둘 다, 단 다른 의미로

요청에는 **두 시점**이 있어서 주체가 갈린다:

| 행위 | 담당 | 비유 |
|------|------|------|
| 요청을 **정의/구성** ("뭘 시킬지") | **Client** | 리모컨 버튼에 기능을 *프로그래밍하는 사람* |
| 요청을 **발동** ("지금 실행해") | **Invoker** | 리모컨 *버튼을 누르는 행위* |
| 요청을 **수행** ("실제로 일함") | **Receiver** | 전등 자체 |

```java
remote.setCommand(new TurnOnLightCommand(livingRoomLight));  // Client: 정의
remote.buttonPressed();                                       // Invoker: 발동
```

### Invoker의 정체성

> Invoker는 *"요청을 실행하는 행위 자체를 관리하는 자"* 다.
> 무슨 요청인지(Command), 누가 수행하는지(Receiver)는 모른 채, **오직 "실행이라는 사건"을 통제**한다.

가장 단순한 Invoker는 그냥 즉시 실행만 한다(이 단계에선 거의 무의미해 보인다). Invoker의 존재 이유는
**"실행 사이에 끼어드는 상위 로직"**(로깅·히스토리·undo·큐잉·타이밍)이 생길 때 폭발한다.

## 3. 예제 — Lab12 완전 코드

```java
// ICommand.java
public interface ICommand {
    void execute();
}

// Light.java (Receiver)
public class Light {
    public void turnOn()  { System.out.println("Light is ON"); }
    public void turnOff() { System.out.println("Light is OFF"); }
}

// Fan.java (Receiver)
public class Fan {
    public void start() { System.out.println("Fan started"); }
    public void stop()  { System.out.println("Fan stopped"); }
}

// TurnOnLightCommand.java (ConcreteCommand)
public class TurnOnLightCommand implements ICommand {
    private Light light;
    public TurnOnLightCommand(Light light) { this.light = light; }
    @Override public void execute() { light.turnOn(); }
}

// TurnOffLightCommand.java
public class TurnOffLightCommand implements ICommand {
    private Light light;
    public TurnOffLightCommand(Light light) { this.light = light; }
    @Override public void execute() { light.turnOff(); }
}

// StartFanCommand.java
public class StartFanCommand implements ICommand {
    private Fan fan;
    public StartFanCommand(Fan fan) { this.fan = fan; }
    @Override public void execute() { fan.start(); }
}

// StopFanCommand.java
public class StopFanCommand implements ICommand {
    private Fan fan;
    public StopFanCommand(Fan fan) { this.fan = fan; }
    @Override public void execute() { fan.stop(); }
}

// HomeAutomationRemote.java (Invoker)
public class HomeAutomationRemote {
    private ICommand command;
    public void setCommand(ICommand command) { this.command = command; }
    public void buttonPressed() { command.execute(); }
}
```

> `HomeAutomationRemote`에는 `Light`나 `Fan`이라는 단어가 한 번도 안 나온다.
> 새 장치를 100개 추가해도 Invoker는 수정 불필요 → OCP 준수.

## 4. 심화

### 4.1 Undo/Redo (가장 강력한 변형)

```java
public interface ICommand {
    void execute();
    void undo();          // 추가
}

public class TurnOnLightCommand implements ICommand {
    private Light light;
    private boolean previousState;   // 이전 상태 저장
    public TurnOnLightCommand(Light light) { this.light = light; }
    @Override public void execute() {
        previousState = light.isOn();
        light.turnOn();
    }
    @Override public void undo() {
        if (!previousState) light.turnOff();
    }
}
```

Invoker가 실행한 명령을 스택에 쌓는다 → 포토샵 Ctrl+Z, Word 실행취소, 게임 수 되돌리기의 핵심.

### 4.2 Macro Command (복합 명령)

```java
public class MacroCommand implements ICommand {
    private List<ICommand> commands;
    public MacroCommand(List<ICommand> commands) { this.commands = commands; }
    @Override public void execute() {
        for (ICommand cmd : commands) cmd.execute();
    }
}
```

### 4.3 기타 변형
- **Queueing**: 명령을 큐에 쌓아 순차 처리 (프린터 스풀러, Jenkins, RabbitMQ)
- **Logging/Transaction**: 명령을 디스크에 기록 → 크래시 복구 (DB의 WAL)

### 4.4 함수형 치환
함수가 1급 객체인 언어(Python/JS)에서는 Command가 그냥 함수/람다로 끝난다.

```java
// Java 8+ 람다 — ICommand는 추상 메서드 1개(함수형 인터페이스)라 람다 가능
ICommand cmd = () -> light.turnOn();
```

단, **상태(undo용 이전값)** 나 **메서드 2개 이상(execute+undo)** 이 필요하면 클래스가 유리하다.
(추상 메서드가 2개면 함수형 인터페이스가 아니라 람다 불가)

### 4.5 다른 패턴과의 관계

| 패턴 | Command와의 관계 |
|------|----------------|
| Composite | MacroCommand는 Composite의 합성 아이디어를 *차용* (정석 적용은 아님) |
| Memento | undo 구현 시 이전 상태 저장에 Memento 활용 |
| Strategy | 구조 유사. 차이는 "Strategy=알고리즘 교체", "Command=요청 캡슐화" |

### 4.6 안티패턴
- ❌ Receiver 없이 Command가 모든 로직을 직접 가짐 → Command 비대화. **Command는 얇은 래퍼여야.**
- ❌ 단순 CRUD에 Command 남발 → 클래스 폭발. 람다로 충분.

**도입 기준**: Undo/Redo · 큐잉 · 로깅/직렬화 · 매크로 · 시간 분리 중 하나라도 필요할 때.

## 📌 Command 학습 Q&A 정리

**Q: MacroCommand는 Composite인가?**
A: 정석 적용이 아니라 **Composite의 합성 원리를 차용한 단순화 버전**. 대부분의 MacroCommand는
트리가 아니라 평평한 리스트이고, 구조가 동적으로 잘 안 바뀌며, add/remove 같은 구조 관리 연산이
거의 없다. Composite의 진짜 가치(재귀 트리 + 런타임 유동성)는 약하다. GoF도 "Composite로 구현
*가능*"이라고 가능성으로만 언급한다.

**Q: Command undo와 Memento의 차이?**
A: 저장하는 *대상*이 다르다.
- **Command undo**: **연산(operation)** 자체를 기억 → 반대 연산을 수행 (켰으면 끈다)
- **Memento**: **상태(state)** 스냅샷을 저장 → 통째로 복원
- 자주 함께 쓴다: Command가 "언제 저장/복원할지(흐름)"를, Memento가 "무엇을 어떻게 저장할지(상태)"를 담당.

**Q: 1급 객체와 람다?**
A: 1급 객체 = 변수에 담고/인자로 넘기고/반환할 수 있는 대상. 전통 Java는 함수가 1급이 아니어서
"함수를 객체로 포장"하려고 Command가 등장한 면이 있다. Java 8 람다는 **함수형 인터페이스(추상
메서드 1개)** 타입으로 함수를 값처럼 다루게 해준다. `ICommand`가 정확히 함수형 인터페이스.

**Q: MacroCommand와 Invoker의 차이?**
A: 칼로 잘린다. **"이게 ICommand인가?"** 로 구분.
- MacroCommand: ✅ ICommand다. 여러 명령을 묶은 *하나의 명령* (WHAT)
- Invoker: ❌ ICommand 아니다. 명령을 들고 실행을 *관리하는 외부 존재* (HOW)
- 비유: MacroCommand=플레이리스트, Invoker=플레이어. 플레이어는 자기가 트는 게 한 곡인지 플레이리스트인지 모른다.

# 패턴 2: Decorator 패턴

## 1. 개념

**한 줄 정의**: 객체를 같은 타입의 다른 객체로 감싸서, 원본을 건드리지 않고 기능을 동적으로 덧붙인다.

### 문제 상황 — 상속의 한계 (커피 옵션)

상속으로 옵션 조합을 풀면 옵션 n개에 대해 2ⁿ개의 클래스가 필요(**클래스 폭발**). 게다가 상속은
**컴파일 타임에 고정**되어, 런타임에 손님이 옵션을 고르는 동적 결정을 표현할 수 없다.

### 핵심 아이디어

> **상속(extends) 대신 합성(composition)으로 기능을 쌓는다. 감싸고, 또 감싼다 (마트료시카).**

```java
Coffee coffee = new BasicCoffee();
coffee = new MilkDecorator(coffee);
coffee = new SyrupDecorator(coffee);
```

핵심: **감싸는 객체(Decorator)도 감싸지는 객체와 같은 타입**이다.

## 2. 구조 — 4가지 역할 + 이중성

| 역할 | 책임 | Lab12 | 커피 |
|------|------|-------|------|
| **Component** | 공통 인터페이스 | `Shape` | `Coffee` |
| **ConcreteComponent** | 감싸질 원본 | `Circle`, `Rectangle` | `BasicCoffee` |
| **Decorator** | Component를 *구현* + Component를 *필드로 보유* | `ShapeDecorator` | `CoffeeDecorator` |
| **ConcreteDecorator** | 실제 기능 덧붙임 | `RedShapeDecorator` | `MilkDecorator` |

### 결정적 포인트 — Decorator의 이중성

```java
public abstract class ShapeDecorator implements Shape {  // ① Shape이다 (IS-A)
    protected Shape decoratedShape;                       // ② Shape을 가진다 (HAS-A)
    public ShapeDecorator(Shape decoratedShape) {
        this.decoratedShape = decoratedShape;
    }
}
```

"Shape인 동시에 Shape을 품은 것" = 무한 중첩 가능한 껍질. 이것이 마트료시카가 가능한 이유.

## 3. 예제 — Lab12 완전 코드

```java
// Shape.java
public interface Shape { void draw(); }

// Circle.java
public class Circle implements Shape {
    @Override public void draw() { System.out.println("Shape: Circle"); }
}

// Rectangle.java
public class Rectangle implements Shape {
    @Override public void draw() { System.out.println("Shape: Rectangle"); }
}

// ShapeDecorator.java (추상 Decorator)
public abstract class ShapeDecorator implements Shape {
    protected Shape decoratedShape;
    public ShapeDecorator(Shape decoratedShape) { this.decoratedShape = decoratedShape; }
    @Override public void draw() { decoratedShape.draw(); }  // 기본은 위임
}

// RedShapeDecorator.java (ConcreteDecorator)
public class RedShapeDecorator extends ShapeDecorator {
    public RedShapeDecorator(Shape decoratedShape) { super(decoratedShape); }
    @Override public void draw() {
        decoratedShape.draw();        // ① 원래 동작 위임
        setRedBorder(decoratedShape); // ② 내 기능 추가
    }
    private void setRedBorder(Shape s) { System.out.println("Border Color: Red"); }
}
```

실행 결과:
```
redCircle.draw();
// Shape: Circle
// Border Color: Red
```

## 4. 심화

### 4.1 여러 겹 쌓기 — 재귀적 풀림

```java
Beverage order = new Espresso();
order = new Milk(order);
order = new Syrup(order);
order = new Syrup(order);   // 시럽 두 번! 상속으론 불가능
// cost() 호출 시: 바깥 껍질이 안쪽에게 묻고, 그 답에 자기 몫을 더해 돌려줌 (재귀)
```

### 4.2 개입 방식 3가지
- **before**: 원본 호출 전 처리
- **after**: 원본 호출 후 처리 (Lab의 RedShapeDecorator)
- **around**: 원본 결과를 가공 (예: `cost() * 1.1`)

### 4.3 실무 — Java I/O 스트림 (대표 사례)

```java
InputStream in = new FileInputStream("data.txt");  // ConcreteComponent
in = new BufferedInputStream(in);                    // 버퍼링 추가
in = new GZIPInputStream(in);                        // 압축 해제 추가
```

전부 `InputStream` 타입이라 무한 조합 가능. 단, 위 예시 자체는 컴파일 타임에 박은 것이라
"동적"의 진면목은 조건부 조립에서 드러난다:
```java
if (config.isCompressed()) in = new GZIPInputStream(in);  // 런타임 결정
```

### 4.4 함수형 Decorator (변형 A)

리턴 타입을 원본 타입으로 하면 함수 합성으로 데코레이터를 만들 수 있다:
```java
Shape decorateRed(Shape s) {
    return () -> { s.draw(); System.out.println("Red"); };  // 입력도 Shape, 출력도 Shape
}
Shape result = decorateBold(decorateRed(new Circle()));
```
`Shape → Shape` 함수가 "Shape를 받아 Shape를 돌려주는 껍질" 역할. 단일 메서드 + 상태 없을 때 깔끔.

### 4.5 동적 프록시 (변형, 가장 강력)

데코레이터를 컴파일 타임에 손으로 안 짜고 **런타임에 자동 생성**(`java.lang.reflect.Proxy`, CGLIB).
모든 메서드 호출을 핸들러로 가로채 공통 기능 주입 → **AOP의 토대**. Spring `@Transactional`, `@Cacheable`이 이 방식.
- **강력함과 위험함이 동전의 양면**: 컴파일러가 못 막음(리플렉션, 런타임 에러), "마법"이 되어 디버깅 난해, 성능 비용.
- 그래서 실무는 직접 안 짜고 검증된 프레임워크(Spring)에 위임.

### 4.6 헷갈리는 4형제 — 구조는 닮고 의도가 다름

| 패턴 | 구조 | 의도 | 핵심 차이 |
|------|------|------|----------|
| **Decorator** | 같은 타입으로 감쌈 | 기능 추가 | 인터페이스 그대로, 책임만 늘림 |
| **Proxy** | 같은 타입으로 감쌈 | 접근 제어 | 캐싱·지연로딩·권한 (기능 추가 아님) |
| **Adapter** | 다른 타입으로 감쌈 | 인터페이스 변환 | 호환 안 되는 둘을 연결 |
| **Composite** | 같은 타입을 *여러 개* 품음 | 부분-전체 트리 | 1개가 아니라 N개 자식 |

- Decorator vs Composite: `Shape decoratedShape`(단수) vs `List<Node> children`(복수)
- `Collections.synchronizedList(list)`: 기존 리스트를 감싸 동기화 추가 → Decorator의 *느슨한 차용*

### 4.7 안티패턴
- ❌ 순서 의존성: 압축→암호화 vs 암호화→압축은 결과가 다름
- ❌ 타입 식별 깨짐: `instanceof Espresso`가 감싸지면 false
- ❌ 너무 많은 작은 데코레이터 → 수십 겹 쌓여 디버깅 악몽

## 📌 Decorator 학습 Q&A 정리

**Q: 타입 문제는 데코레이터에 원본 타입을 적어 해결하면 되지 않나?**
A: 잘 안 된다. 여러 겹 쌓으면 "원본"이 모호해지고(Milk가 품은 건 Syrup이지 Espresso가 아님),
타입을 추적하기 시작하면 **데코레이터의 투명성(겹을 신경 안 써도 됨)이라는 장점 자체가 깨진다.**
"타입 식별이 필요하다는 것 자체가 데코레이터가 안 맞는다는 신호." 다형성으로 풀 문제다.

**Q: Java I/O가 컴파일 레벨에서 정해진 예시라 애매하다.**
A: 정확한 관찰. "동적"에는 두 층위가 있다 — ① 조합을 누가 언제 결정하는가(예시 코드는 정적), ②
구조가 동적 조합을 *허용*하는가(타입 시스템상 자유로움). I/O의 가치는 ②. 상속이었다면
`BufferedGZIPFileInputStream` 같은 조합 클래스를 다 만들어야 했을 것. **구조와 사용을 분리해서 봐야 한다.**

**Q: 한계 보완과 별개로 변형이 있나?**
A: (A) 구조 단순화 — 추상 Decorator 생략(데코레이터 1개일 때), 함수형(함수 합성), 동적 프록시(런타임 생성→AOP).
(B) 사촌 — Interceptor/Middleware(데코레이터 × CoR 하이브리드), Wrapper(synchronizedList 등 느슨한 차용).
**가장 중요한 건 동적 프록시** — "직접 짜는" 단계를 넘어 "프레임워크가 런타임에 감싸주는" 진화형.

# 패턴 3: Iterator 패턴

## 1. 개념

**한 줄 정의**: 컬렉션의 내부 구조를 노출하지 않고, 그 안의 요소들을 순서대로 접근하는 방법을 제공한다.

### 문제 상황

```java
for (int i = 0; i < shelf.getLength(); i++) {
    Book book = shelf.getBookAt(i);   // 배열·인덱스 구조에 의존
}
```

"어떻게 저장돼 있는가(내부 구조)"가 "어떻게 순회하는가(클라이언트 코드)"로 **새어나왔다**.
내부를 `LinkedList`/`HashMap`으로 바꾸면 깨진다.

### 핵심 아이디어

> **"순회하는 책임"을 컬렉션에서 분리해서 별도 객체(Iterator)에게 맡긴다.**

```java
Iterator<Book> it = shelf.iterator();
while (it.hasNext()) { Book book = it.next(); ... }  // hasNext / next 두 질문만
```

## 2. 구조 — 4가지 역할

| 역할 | 책임 | Lab12 |
|------|------|-------|
| **Iterator** (인터페이스) | `hasNext()`, `next()` | `Iterator<E>` (자바 표준) |
| **ConcreteIterator** | 순회 위치(상태) 관리 | `BookShelfIterator` |
| **Aggregate/Iterable** | iterator 생성 메서드 선언 | `Iterable<E>` |
| **ConcreteAggregate** | 요소 보유, iterator 생성 | `BookShelf` |

### 결정적 포인트 — 상태는 Iterator가 들고 있다

```java
public class BookShelfIterator implements Iterator<Book> {
    private int index;   // "지금 몇 번째" — 이 상태가 Iterator에 있음
}
```

상태를 Iterator가 들고 있어 **한 컬렉션을 동시에 여러 방식으로 순회** 가능(A는 앞에서, B는 뒤에서).
컬렉션이 상태를 들고 있었다면 동시 순회가 서로 간섭했을 것.

(C++ STL의 iterator는 **포인터를 일반화**한 것. raw 포인터도 가장 단순한 iterator. list/map은 메모리상
연속이 아니라 포인터 `++`로 못 가지만, iterator는 각 컨테이너에 맞는 `++`를 숨겨 통일된 방식 제공.)

## 3. 예제 — Lab12 완전 코드

```java
// Book.java
public class Book {
    private String name;
    public Book(String name) { this.name = name; }
    public String getName() { return name; }
}

// BookShelf.java (ConcreteAggregate)
import java.util.Iterator;
public class BookShelf implements Iterable<Book> {
    private Book[] books;
    private int last = 0;
    public BookShelf(int maxsize) { this.books = new Book[maxsize]; }
    public Book getBookAt(int index) { return books[index]; }
    public void appendBook(Book book) { this.books[last] = book; last++; }
    public int getLength() { return last; }
    @Override public Iterator<Book> iterator() { return new BookShelfIterator(this); }
}

// BookShelfIterator.java (ConcreteIterator)
import java.util.Iterator;
public class BookShelfIterator implements Iterator<Book> {
    private BookShelf bookShelf;
    private int index;
    public BookShelfIterator(BookShelf bookShelf) { this.bookShelf = bookShelf; this.index = 0; }
    @Override public boolean hasNext() { return index < bookShelf.getLength(); }
    @Override public Book next() {
        Book book = bookShelf.getBookAt(index);
        index++;
        return book;
    }
}
```

> `for-each`는 컴파일러가 `iterator()` + `hasNext`/`next`로 **자동 변환**한다.
> `Iterable`만 구현하면 for-each가 공짜 → Iterator 패턴이 언어에 내장된 모습.

## 4. 심화

### 4.1 외부 반복자 vs 내부 반복자 — 누가 제어하는가
- **외부(External)**: 클라이언트가 `next()`를 직접 호출 (Lab12 방식). 중간 멈춤·건너뛰기·동시 순회 가능.
- **내부(Internal)**: 컬렉션에 동작(람다)을 넘김 (`forEach`). 간결하나 세밀한 제어 어려움. 자바 Stream이 내부 반복자.
- 둘은 배타적이지 않다. `List`는 `iterator()`(외부)와 `forEach()`(내부)를 다 제공.

### 4.2 게으른 평가 (Lazy Evaluation) — 진짜 힘

```java
public class NaturalNumberIterator implements Iterator<Integer> {
    private int current = 0;
    public boolean hasNext() { return true; }      // 무한
    public Integer next() { return ++current; }    // 호출 시 즉석 생성
}
```

전체를 메모리에 안 올리고 **필요할 때 하나씩 생성** → 무한 수열·파일 한 줄씩·DB 한 행씩(ResultSet)·스트림 처리.

### 4.3 Generator — 문법적 진화
Python `yield`, JS `function*`는 `hasNext`/`next`를 손으로 안 짜도 되게 해줌(상태를 언어가 관리).
자바는 generator 문법이 없어 클래스로 직접 작성(Lab 방식). **개념 동일, 문법만 진화.**

### 4.4 fail-fast
순회 중 컬렉션을 수정하면 `ConcurrentModificationException`. "조용히 틀린 결과보다 즉시 터뜨려 버그를 빨리"라는 철학(`modCount` 체크). C++은 iterator 무효화(invalidation)로 댕글링 위험.

### 4.5 다양한 순회 전략
트리는 preOrder/inOrder/postOrder/BFS 등 여러 Iterator 제공 가능 → 순회 알고리즘 캡슐화(Strategy와 사촌).

### 4.6 다른 패턴과의 관계
- **Composite**: 트리 순회 시 Iterator가 짝
- **Factory Method**: `iterator()`가 사실 Factory Method
- **Visitor**: Iterator는 "순회", Visitor는 "각 요소에 연산" → 짝

### 4.7 안티패턴
- ❌ Iterator로 컬렉션 수정 (읽기 순회용)
- ❌ 단순한 경우 과한 Iterator
- ❌ 내부 구조 노출(`getBooks()`로 배열 통째 반환) → 캡슐화 무의미

## 📌 Iterator 학습 Q&A 정리

**Q: STL/컬렉션의 iterator가 이 패턴인가?**
A: 맞다. C++ `*it`/`++it`, 자바 `hasNext`/`next`는 표현만 다른 같은 패턴. iterator는 **포인터의 일반화**.

**Q: 이 패턴은 순회만 위해 존재하나?**
A: 책임은 순회 하나(SRP). 하지만 "순회"가 넓다 — 데이터 소스 추상화(파일·DB·네트워크를 같은 방식으로),
게으른 생성, 파이프라인(Stream/ranges/generator)의 토대. 비유: 수도꼭지(뒤에 물탱크든 강이든 똑같이 틀어 씀).

**Q: ADT의 "현재 인덱스 변수"를 분리한 게 핵심인가?**
A: 정확. 더 일반화하면 **"순회 상태의 분리"**(트리는 인덱스가 아니라 경로 스택). 분리 덕에 동시 다중 순회 가능.

**Q: 컨테이너가 바뀌면 무엇이 안 바뀌나?**
A: ConcreteAggregate/ConcreteIterator는 짝꿍이라 함께 **바뀐다.** 안 바뀌는 건 **Iterator 인터페이스 + 클라이언트**.
(DIP: 클라이언트가 추상에 의존). 비유: 콘센트 모양(인터페이스)은 표준이라 안 바뀌고, 뒤 배선(ConcreteIterator)은 건물(Aggregate)마다 다름.

**Q: 순회+검색이 둘 다 필요하면?**
A: Iterator는 "전체를 대체하는 패턴"이 아니라 "한 책임을 담당하는 부품". 컬렉션은 자기 책임(추가·삭제·검색)을
다 가지면서 **순회만 Iterator로 위임**한다. 검색은 자료구조 강점을 살린 전용 메서드로(HashMap의 `get`은 O(1),
순회로 검색하면 O(n)이라 해시 의미가 사라짐). 순회와 나머지 연산은 분업한다.

**Q: 언어 내장 generator의 부작용?**
A: ① 일회성(소진) — 한 번 돌면 끝, 다시 못 돔. ② 지연 실행 — 에러가 정의한 곳이 아니라 순회하는 곳에서 터짐.
③ 순회 중 수정(fail-fast/무효화). "강력하되 함정을 알고 써야 함." 단순·직관적 순회에 광범위하게 쓰되 복잡한 제어는 직접 구현.

**Q: fail-fast 대신 스냅샷으로?**
A: 실재하는 해법. `CopyOnWriteArrayList`(copy-on-write)가 그것. 본질은 **일관성 vs 최신성 트레이드오프**:
fail-fast(터뜨림)·락(얼림)·스냅샷(과거를 일관되게 봄). "실시간 시점 순회"는 본질적으로 모순적이라 처리하기 나름.
(이름은 Memento보다 "스냅샷/copy-on-write"가 정확 — Memento는 undo 목적, 여기선 안정된 뷰 확보 목적.)

# 패턴 4: Visitor 패턴

## 1. 개념

**한 줄 정의**: 데이터 구조(요소들)는 그대로 두고, 그 요소들에 수행할 연산을 별도 "방문자(Visitor)"로 분리한다.

### 문제 상황

도형(Circle, Rectangle...)에 연산(넓이·그리기·XML·둘레...)을 추가할 때, 각 도형 클래스에 메서드를 다 넣으면:
1. 새 **연산** 추가 시 *모든* 도형 클래스를 수정
2. 도형 클래스가 무관한 책임(XML, 렌더링)으로 비대 (SRP 위반)
3. 연산 코드가 여러 클래스에 흩어짐

### 핵심 아이디어

> **연산을 요소에서 떼어내, "방문자(Visitor)"라는 별도 객체에 모은다.**

```java
circle.accept(new AreaVisitor());   // 원의 넓이 계산
circle.accept(new XmlVisitor());    // 원을 XML로
```

새 연산 추가 시 도형을 안 건드림. 새 Visitor 클래스 하나만 만들면 끝.

## 2. 구조 — 이중 디스패치 (이 패턴의 심장)

### 단일 디스패치 vs 이중 디스패치

자바의 일반 다형성은 **단일 디스패치** — `obj.method()`에서 실행 메서드는 **`obj`의 타입 하나**로 결정.

Visitor는 **두 타입**(요소 타입 × 연산 타입)에 따라 동작이 달라져야 한다. 단일 디스패치로는 한 타입만 본다.

**이중 디스패치** = 다형성 호출을 두 번 연쇄:

```java
circle.accept(visitor);          // 1차: circle 타입(Circle) 확정
// Circle.accept 내부:
public void accept(Visitor v) {
    v.visitCircle(this);          // 2차: visitor 타입 확정 + this는 Circle로 확정
}
```

두 번의 다형성으로 "Circle × AreaVisitor" 조합이 정확히 찍힌다. 핵심은 `accept` 안에서 `this`(타입 확정된 요소)를 visitor에게 도로 넘기는 것.

## 3. 예제

```java
// Element 쪽 (구조)
public interface Shape { void accept(Visitor visitor); }

public class Circle implements Shape {
    double radius = 5;
    @Override public void accept(Visitor visitor) { visitor.visitCircle(this); }
}
public class Rectangle implements Shape {
    double width = 4, height = 6;
    @Override public void accept(Visitor visitor) { visitor.visitRectangle(this); }
}

// Visitor 쪽 (연산)
public interface Visitor {
    void visitCircle(Circle c);
    void visitRectangle(Rectangle r);
}
public class AreaVisitor implements Visitor {
    public void visitCircle(Circle c) {
        System.out.println("Circle area: " + (Math.PI * c.radius * c.radius));
    }
    public void visitRectangle(Rectangle r) {
        System.out.println("Rectangle area: " + (r.width * r.height));
    }
}
public class XmlVisitor implements Visitor {  // 도형 안 건드리고 연산 추가!
    public void visitCircle(Circle c)    { System.out.println("<circle r=\"" + c.radius + "\"/>"); }
    public void visitRectangle(Rectangle r) { System.out.println("<rect .../>"); }
}

// 사용 — Iterator와 결합
List<Shape> shapes = List.of(new Circle(), new Rectangle());
Visitor area = new AreaVisitor();
for (Shape s : shapes) {   // Iterator로 순회
    s.accept(area);        // 각 요소에 Visitor 연산 적용
}
```

## 4. 심화

### 4.1 OCP의 비대칭 — 이 패턴의 전부 (표현 문제, Expression Problem)

| 무엇을 추가? | Visitor에서 | OCP? |
|---|---|---|
| **새 연산** (JSON 출력 등) | 새 Visitor 클래스 하나 | ✅ 천국 |
| **새 요소 타입** (Triangle 등) | *모든* Visitor에 visitTriangle 추가 | ❌ 지옥 |

> **판단 기준**: 요소 타입이 안정적 + 연산이 자주 늘면 → Visitor 천국.
> 요소 타입이 자주 늘면 → Visitor 쓰지 마라.

Decorator(한 축만 있어 진짜 천국)와 달리, Visitor는 한 축 천국 + 직교 축 지옥. 그래서 "쓸지 말지"가 신중해야 함.

### 4.2 왜 이중 디스패치인가 — instanceof의 대안

```java
// 안티패턴: instanceof 사슬
if (s instanceof Circle) { ... }
else if (s instanceof Rectangle) { ... }
return 0;  // 새 도형 빠뜨리면 조용히 0 (런타임 버그)
```

instanceof는 새 타입 누락을 **컴파일러가 못 잡는다**(조용한 런타임 버그). Visitor는 인터페이스에
`visitTriangle`을 추가하면 **모든 Visitor 구현체가 컴파일 에러** → 누락을 **컴파일 타임에 강제로** 잡음.
즉 **타입 분기를 컴파일 안전하게 객체지향으로 푸는 방법.** (단, 이 강제가 4.1 "새 요소=지옥"의 대가.)

### 4.3 함수형 — 패턴 매칭

```java
// Java 21+ 패턴 매칭 switch
double area = switch (shape) {
    case Circle c    -> Math.PI * c.radius * c.radius;
    case Rectangle r -> r.width * r.height;
};
```

Visitor와 패턴 매칭은 **표현 문제의 같은 쪽을 푸는 두 방법**(OOP vs 함수형). 트레이드오프 구조 동일.
자바 패턴 매칭 도입으로 Visitor의 장황한 보일러플레이트 없이 같은 일 가능.

### 4.4 실무 — 컴파일러 AST (대표 사례)

AST 노드(IfNode, WhileNode...)는 고정적(언어 문법 안정), 연산(타입검사·최적화·코드생성)은 계속 늘어남
→ **요소 안정 + 연산 증가 = Visitor 천국**. 거의 모든 컴파일러가 AST를 Visitor로 처리.
자바 `Files.walkFileTree`의 `FileVisitor`도 표준 Visitor(순회=Iterator + 연산=Visitor).

### 4.5 Composite와의 단짝
Composite 트리에 연산 적용 시 강력. **Composite(구조) + Iterator(순회) + Visitor(연산)** 삼각 협력.

### 4.6 안티패턴
- ❌ 요소가 자주 바뀌는데 Visitor 사용 (4.1 지옥)
- ❌ 캡슐화 침해 — Visitor가 요소 내부를 봐야 해서 getter를 강요
- ❌ 단순한 경우 오버엔지니어링

## 📌 Visitor 학습 Q&A 정리

**Q: Visitor는 OCP를 해결한 게 아니라 연산객체로 떠넘긴 것 아닌가?**
A: 절반 이상 맞다. **책임의 총량은 안 줄고 위치만 바뀐다.** Visitor가 한 일은 **변경의 축을 회전**시킨 것:
연산 추가는 천국으로, 요소 추가는 지옥으로. "OCP 위반의 고통을 *내가 덜 자주 건드리는 축*으로 몰아넣는"
전략적 떠넘기기다. 표현 문제는 이론적으로도 "양쪽 축을 동시에 열 수 없음"이 증명돼 있어, Visitor는 부분해.
"최적화 수준"이라는 인상이 정확하다.

**Q: 팩토리+브릿지로 연산객체 만들어 accept로 받으면 Visitor를 대체?**
A: 보완은 되지만 **이중 디스패치(심장)는 대체 못 한다.**
- 팩토리: Visitor *생성*을 도움 (직교)
- 브릿지: Visitor *내부 연산의 구현 분리*를 도움 (한 단계 안쪽)
- `accept` 안의 `v.visit(this)`라는 "요소가 자기를 확정해 도로 넘기는" 그 순간은 사라지지 않는다.

**Q: 그냥 연산객체 만들어 accept로 받고 안에서 처리하면?**
A: 두 갈래뿐이다.
- **갈래 A** (execute(Shape) 하나): 안에서 `instanceof`로 타입을 다시 까야 함 → 회피하려던 문제로 회귀
- **갈래 B** (execute가 타입별): instanceof는 사라지나 **이게 곧 이중 디스패치** → Visitor 재발명
"이중 디스패치를 안 쓰고 깔끔하게"는 불가능. 타입 분기는 어디엔가 반드시 있고, **"어디에 얼마나 안전하게 두느냐"** 만 선택 가능.

**Q: 브릿지로 도형종류를 구현체로? / 연산구현체를 필드로?**
A: 방향1(연산←도형종류)은 도형이 하는 게 없어 "지옥"(브릿지 형태만 빌린 instanceof). 방향2(도형←연산구현체)는
그럴듯하나 연산 구현체가 모든 도형을 알아야 해 OCP 자유롭지 못함(표현 문제 잔존). **메서드 단위 이중 디스패치가
가장 예쁜 이유**: 타입 누락을 *컴파일러가 강제*(필드 주입은 런타임에야 발견).

**Q: instanceof 강제 디스패치 대체가 무슨 뜻?**
A: "버그를 언제 발견하느냐"의 문제. instanceof는 새 타입 누락을 런타임에 조용히(return 0) 흘려보내지만,
Visitor는 인터페이스에 메서드 추가 시 모든 구현체가 컴파일 에러 → 시끄럽게 컴파일 타임에 강제 검출.
"체크리스트를 손으로 관리" vs "항목 안 채우면 문이 안 열림."

**Q: getter면 캡슐화 OK 아닌가?**
A: 접근 제어 수준에선 public 필드보다 낫다. 하지만 캡슐화의 본질은 "필드 숨김"이 아니라 **"데이터+행동을 한 곳에"**
(Tell, Don't Ask). Visitor는 본질적으로 데이터를 밖(Visitor)으로 꺼내 처리 → Tell, Don't Ask를 위반하는 방향.
getter 자체가 나빠서가 아니라, **Visitor가 getter를 많이 쓰게 만드는 것 = 행동이 데이터에서 떨어져 나간 증거.**
Visitor를 쓰는 순간 캡슐화(데이터+행동 응집)를 일부 포기하기로 선택한 것.

# 종합 비교 및 통찰

## 4개 패턴 한눈에 비교

| | Command | Decorator | Iterator | Visitor |
|---|---|---|---|---|
| **분리하는 것** | 요청(발동↔수행) | 기능(원본↔확장) | 순회(보관↔순회) | 연산(데이터↔행동) |
| **핵심 메커니즘** | 요청 객체화 | 같은 타입 중첩 | 상태 분리 | 이중 디스패치 |
| **OCP 효과** | 새 명령 추가 자유 | 새 기능 추가 자유(천국) | 새 컨테이너 자유 | 새 연산 천국 / 새 타입 지옥 |
| **함수형 치환** | 람다 | 함수 합성 | generator/yield | 패턴 매칭 |
| **대표 실무** | GUI Action, 작업 큐, undo | Java I/O, AOP | STL/컬렉션, Stream | 컴파일러 AST |
| **단짝 패턴** | Composite, Memento | Proxy/Adapter와 구분 | Composite, Visitor | Composite, Iterator |

## Composite-Iterator-Visitor 삼각 관계

> - **Composite**: 일반 노드와 서브트리를 한 타입으로 다룸 (구조)
> - **Iterator**: 트리를 순회 (순회)
> - **Visitor**: 노드와 서브트리에 연산을 일괄 적용 (연산)
>
> 컴파일러 AST, 파일시스템(Lab8 2-2) 등에서 셋이 함께 등장한다.
> **구조 · 순회 · 연산의 삼분할.**

## 관통하는 메타 통찰

**1. 행위 패턴 = 함수.** Command(execute), Iterator(next 생성), Visitor(연산), Strategy(알고리즘)는
모두 "하나의 동작을 객체로 포장한 것". 함수가 1급인 언어에선 포장이 불필요 → 람다·generator·패턴매칭으로 흡수.
구조 패턴(Composite, Bridge)·생성 패턴은 "관계와 형태"를 다뤄 함수로 안 녹는 것과 대조적.

**2. 패턴은 객체지향의 자랑이 아니라 한계를 메우는 반창고.** 상속의 경직성(→Decorator), 단일 디스패치의
한계(→Visitor), 캡슐화와 연산추가의 충돌(→Visitor)을 메운다. 패턴이 23개 존재한다는 것 자체가 메울 구멍이
많다는 증거. 함수형 기능이 들어오면 상당수가 녹는다. **도구의 한계를 보는 건 도구를 진짜 이해했다는 신호.**

**3. 책임의 총량은 보존된다.** 패턴은 책임·보일러플레이트를 *없애는* 게 아니라 *재배치*한다. Visitor의
"OCP 떠넘김", "보일러플레이트 위치 전환"이 그 예. 줄어드는 건 *반복되던 분기(지역 보일러플레이트)*이고,
느는 건 *구조를 위한 골격*. 패턴의 가치는 "더 다루기 좋은 위치로 모으는 것".

**4. 패턴의 가치는 유연성이 실제로 행사될 때만 회수된다.** Decorator·Iterator·Visitor 모두 "변경/다양성이
실제로 일어날 때만" 이득. 안 그러면 오버엔지니어링. 트레이드오프(순서·타입식별·캡슐화·표현문제)를 알고 선택해야 한다.
