# 디자인 패턴 식별: 데코레이터(Decorator)

## 결론
주어진 코드는 **데코레이터 패턴**이다.

## 역할 매핑

| 클래스 | 역할 | 핵심 동작 |
|---|---|---|
| `C_A` | Component | 공통 인터페이스 (`add()`) |
| `C_B` | ConcreteComponent | 장식될 핵심 객체, `"C_B"` 반환 |
| `C_C` | Decorator (추상) | `C_A` 구현 + `C_A` 필드 보유, 위임 |
| `C_D`, `C_E` | ConcreteDecorator | `super.add()` 후 자기 문구 추가 |

## 판단 근거

1. 자기참조 구조 — `C_C`가 `C_A`를 구현하면서 동시에 `C_A`를 필드로 보유한다. 겉보기 타입을 유지한 채 다른 `C_A`를 감쌀 수 있다. (다이어그램의 구현선 + 집약선 동시 연결)
2. 무한 중첩 — 장식자가 같은 타입을 품으므로 장식자끼리 겹친다. `new C_D(new C_E(new C_B()))`의 결과는 안쪽부터 `"C_B" -> +first → +second` 순으로 쌓인다.
3. 조합 기반 확장 — 조합마다 하위 클래스를 만드는 대신, 런타임에 장식자를 끼워 기능을 누적한다.
