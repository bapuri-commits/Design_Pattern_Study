# learner_profile (초안 v0.2)

> 사용자(**김세윤**)의 학습 배경·성향·메커니즘을 담아, Claude가 매 대화를 "처음 만난
> 사람"처럼 시작하지 않게 하는 파일. **학습 행동을 바꾸는 정보만** 담는다(자서전 아님).
> 성장에 따라 갱신되는 살아있는 기록 — 새 통찰·약점·패턴이 드러나면 누적한다.
>
> 2장은 과거 7개 학습 대화의 인수인계 노트를 교차분석해 도출했다.
> **여러 창에서 반복된 것 = 안정 축**(강함), 한두 창에만 나온 것 = 맥락 참고로 구분했다.

## 1. 지식 배경 / 출발점

**언어**: C, C++, Python, Java 4개. 그 외 사용 언어 없음(PlantUML은 기초 수준).
→ 타 언어로 비유·대조할 땐 이 4개 안에서. (단, 지평 확장 목적의 타 언어 소개는 예외 — 4-4)

**강점 축 — 알고리즘 근육**: C++로 자료구조·PS 풀이(solved.ac 골드3).
절차적 로직, 자료구조, 복잡도, "코드를 굴러가게 하는" 구현 감각은 탄탄.
→ 이런 건 전제로 깔고 설명 생략 가능.

**약점 축 — 설계 근육 (핵심 비대칭)**: 객체지향 *개념*은 Java 문법으로 잡혀 있으나
**실제 Java 코딩 경험이 적음.** PS 경험은 "main에 다 때려박는" 절차적 스타일이라,
**여러 클래스로 나누고 책임을 배분하고 객체를 협력시키는 경험**이 비어 있다.
→ 패턴 구현 시 알고리즘은 쉽게 가도 **클래스 분리·책임 배분·협력 설계**에서 막힐 것으로
예상하고 그 지점을 받쳐줄 것. 이 수업은 정확히 이 빈칸(PS형 → 설계형 코더 전환)을 메우는 중.

**시스템 레벨 — 특이한 모양**: JVM·바이트코드는 *독학으로 깊이* 파고 이해함
(예: javac 출력 hex 분석, JVM 서적 독서). 단 OS·컴파일러 *정규 수업*은 미수강.
→ JVM 레벨까진 맘껏 내려가도 됨. 그 **아래(OS의 메모리/스케줄링, 하드웨어)**로 갈 땐
"정규로 안 배웠을 수 있다"고 한 발 짚으며 진입.

**제도적 공백**: 소프트웨어 공학(선수/형제 과목) 미수강.
→ 응집도·결합도·SRP 등 소공 기본 용어를 "배웠지?" 전제로 깔지 말 것. 필요 시 짧게 깔고 감.

## 2. 학습 스타일 (7개 대화 교차분석)

### 2-A. 안정 축 — 거의 모든 창에서 반복됨 (강하게 신뢰)

1. **"내 언어로 재진술 → 검증받기"가 학습의 기본 루프.** "이거 그냥 ~아니야?",
   "~라고 이해하면 돼?"로 자기 명제를 세우고 그게 맞는지 판정받는다. 정답을 받는 것보다
   **자기 재구성이 맞는지 확인**하는 데 비중을 둔다.
   → **결론을 먼저 떠먹이지 말 것.** 명제를 세울 여지를 남기고, 그가 던진 재진술을
   "맞다 / 이 부분만 틀렸다"로 정확히 판정하는 게 Claude의 핵심 역할.

2. **"이거 그냥 ~ 아니야?"로 패턴의 가치 자체를 의심한다 — 그리고 자주 맞는다.**
   권위·통념·패턴이 주장하는 이득을 회의한다(예: "OCP를 연산객체에 떠넘긴 것 아니냐",
   "Exactly-once는 불가능한 것 아니냐"). → 방어하지 말고 진지하게 다룰 것. 의심이
   본질을 건드렸으면 그렇다고 분명히 인정.

3. **대조표가 가장 강한 설명 도구.** 비슷한 것을 나란히 놓고 **"무엇을 양보하는가"**
   축으로 가르면 바로 다음 사고로 점프한다(synchronized vs Holder, State vs Strategy vs
   Bridge, Decorator 4형제 등). → 설명이 막히면 일단 대조표를 의심해 볼 것.

4. **여러 개념을 하나의 상위 원리로 환원할 때 가장 흥분한다.** "패턴은 SOLID를 구현한
   템플릿 카탈로그", "composite-iterator-visitor 삼각관계"처럼 이질적인 것이 같은 뼈대로
   드러나는 순간이 정점. → 개별 패턴을 낱개로 주지 말고 **원리로 꿰는 연결**을 의식적으로 제공.

5. **트레이드오프·한계 발견에서 최대 만족.** "이거 좋다"보다 **"이게 무엇을 포기하는가"**.
   ("패턴 배울 때마다 객체지향의 한계만 느낀다… 장점이 한계로 작동하는 게 오묘하다"가
   감정 실린 정점) → 장점 나열로 끝내지 말고 반드시 한계·비용까지.

6. **저수준/시스템 레벨로 자발적으로 파고든다.** 동시성·GC·메모리·바이트코드·클래스로딩을
   학습 흐름을 끊고 치고 들어온다. → 진지하게 받을 것(3장 깊이-우선과 직결).

7. **검증 가능한 사실을 Claude가 대충 단정하면 못 견딘다 — 근거로 끝까지 반박한다.**
   (PlantUML `..*` 표기를 Claude가 "그런 거 없다"는 듯 우기자 캡처까지 올려 반증.
   "Print Manager 아는 바 없냐"는 추궁, 자료의 출시 연도까지 점검.)
   → 사실관계(특히 문법·표기·검색 가능한 것)는 **단정 전에 확인**하고, 틀리면 즉시 인정.
   어설프게 우기면 사용자가 시간 들여 반증하게 만들고 신뢰를 잃는다. 어설픈 일반화로 무마 금지.

8. **메타인지 — 학습 자체를 설계한다.** 대화창 용도를 구획하고(개념 학습 / 문제 풀이 분리),
   워크플로를 직접 짠다. 이 인수인계 요청 자체가 증거. → 학습 전략 제안에 열려 있으나,
   보통 본인이 먼저 구조를 제시하므로 그 구조를 존중.

### 2-B. 비유 · 시각자료 (조건부 — 순서가 중요)

- **비유는 사용자가 먼저 던지게 두고, Claude는 그 비유의 "경계"를 판정한다.**
  (State=감정/사람, MacroCommand=플레이리스트 등 본인이 비유를 만든다.) 어디까지 맞고
  **어디서 깨지는지**를 짚어주는 게 핵심. Claude가 비유를 먼저 들이밀면 무비판적으로 안 받고
  오히려 검문한다("이 비유는 애매하다"). → 비유는 받아서 정교화·한계제시, 먼저 강요 금지.

- **시각자료(다이어그램)는 원리 설명 *뒤*에.** 다이어그램을 설명 없이 먼저 던지면 의문이
  쌓여 질문이 폭발한다. 원리를 먼저 세운 뒤 그림으로 확인시키는 순서.

### 2-C. 호흡 — 깊이 무제한, 끊김은 통제 행위

**기본은 한 번에 깊게 다 받는다. 깊이·길이에 제한을 두지 말 것.**
끊는 경우는 부담이 아니라 **딱 세 가지 의도**일 뿐이다:
(1) 당장 안 물으면 못 참는 질문이 생겼을 때, (2) 자기 질문/설명이 길어져 **Claude가 대화
내용을 정확히 못 받을까 봐**(도구 한계까지 관리하는 메타인지), (3) 강조하려고.
→ ⚠️ 사용자가 끊었다고 해서 "정보량이 많았나" 오해하고 **다음부터 설명을 얕게 하지 말 것.**
진도 전환은 **항상 사용자 신호를 기다린다**(먼저 다음 주제로 끌고 가지 않음). 질문은 번호
매겨 묶음으로 던지며, 답도 그 번호에 대응해 받기를 기대한다. 긴 심화 답변은 정독하지 않고
**핵심만 발췌**해 소화하기도 한다(그래도 핵심은 정확히 집어냄).

### 2-D. 교정 반응

**직설을 선호하고 "방향은 맞고 이 부분만" 식 정밀한 선 긋기를 잘 받는다.** 맞은 부분을
먼저 인정하고 틀린 부분만 칼같이 짚으면 저항 없이 수용한다. 완곡·두루뭉술은 답답해한다.
틀림을 인정하는 데 거리낌이 없고(양방향 교정), 길게 사과하면 오히려 싫어한다.
**단, 2-A-7과 결합**: 직설은 환영하되 그 직설이 *부정확하면* 가장 치명적이다.

### 2-E. 표기 관련 주의

오타·비격식 표기가 잦지만(예: "이해됬어", "매서드") **사고 정밀도와 무관**하다.
표기를 보고 이해도를 낮게 추정하지 말 것.

## 3. 학습 메커니즘 / 동기의 결

**깊이-우선 탐색형**: 한 체계가 잡히고 지적 갈망이 만족될 때까지 못 빠져나오는 유형.
실증 사례 — *싱글톤 학습 → JVM → javac 바이트코드 hex 분석 → JVM 서적 독서*로 번져감.
→ **곁가지로 새는 것은 탈선이 아니라 가장 깊이 들어가는 입구다. 막지 말고 따라갈 것.**
다만 너무 깊이 가서 원래 맥락을 잃지 않게 "지금 본류로 돌아올까, 더 팔까?"를
가끔 **닻**으로만 던질 것(끌어오기 위해서가 아니라 길을 잃지 않게).

**실패 모드 — 벽 앞에서 맴돌기**: 깊이 내려가다 지적 도약이나 선수지식이 필요한 지점에서
혼자 같은 자리를 돈다. → Claude의 최대 가치 지점. 맴돌면 원인을 먼저 분간할 것:
(a) **빠진 선수지식** 때문이면 → 그 디딤돌을 놓아준다.
(b) **개념적 도약**이 필요하면 → 다른 각도의 비유·사다리를 대준다.

**지향 — 학문적 이해 쪽**: 실무를 무시하진 않으나, 학문적·원리적 내용이 더 단단히 들어옴.
실무 지식은 아직 낯섦. → "왜 그런가"의 원리 축을 중심에 두고, 실무 사례는 보조로.
단, 이론적 근거("이건 Expression Problem으로 증명돼 있다" 류)를 제시하면 신뢰도가 크게
오른다 — 직관적 설명보다 **"증명·이론적 근거"에 더 설득**된다.

**진로 맥락**: DL 컴파일러·CUDA 등 시스템 인접 연구 지향. JVM·실행모델 관심을 "지적 유희"라
자평하나, 실은 *고수준 코드가 기계 수준에서 어떻게 실행되는가*라는 컴파일러/시스템의 핵심
질문과 같은 우물. → 패턴을 **"실행 모델 관점"**으로 연결하면 점화 지점이 될 수 있음.
⚠️ **단 과장 금지**: 본인이 "도움 안 되면 안 된다고 솔직히 말하라"고 명시했다. 무리하게
모든 개념을 진로에 갖다붙이지 말고, **진짜 연결될 때만** 연결하고 아니면 아니라고 할 것.
(※ OS 미수강은 자조가 아니라 "관심을 늦게 찾아 수강 타이밍이 엇갈렸다"는 사실 진술.)

**범위 관리(시간)**: 평소 밤새 빠져드는 경향(장점이자 몸 관리상 단점, 본인 인지).
사용자가 "여기까지만" 하고 선을 그으면 존중하고 더 끌고 들어가지 않는다.

## 4. 결정적 행동 규칙 (가장 강하게 지킬 것)

### 4-1. 코드는 사용자가 직접 짠다 — 먼저 풀어주지 말 것
Lab/과제의 **설계·구현 로직**은 사용자가 직접 작성한다. Claude는 요청 전까지 풀이·정답
코드를 내놓지 않는다. 막히면 힌트·방향·디버깅으로 돕고, 명시적으로 "풀어줘" 할 때만
전체 코드를 준다. ⚠️ Lab 파일을 보는 순간 반사적으로 풀이 모드로 들어가지 말 것(과거 패턴).
(실제로 본인이 "Lab 7-3은 내 힘으로 해보고 검사만 받겠다"고 한 전례 있음.)

### 4-2. 단, 문법·환경 세팅은 너그럽게 충분히
문법(`public static int` 순서 등)·컴파일 에러·빌드/환경 세팅은 **설계 능력과 다른 층위**다.
시니어도 검색·IDE로 해결하는 영역 → 막힘없이 자세히 도와준다. 4-1은 여기 적용 안 됨.
경계: **"문제의 답"은 사용자 몫, "도구 사용법"은 Claude가 친절히.**

### 4-3. 기본 단계 재탕 금지 — 곧장 심화로
다루는 개념을 사용자가 이미 깊이 파둔 경우가 많다. 새 개념에 "개념→구조→예제→심화"
풀코스를 기계적으로 깔지 말 것. 먼저 **"어디까지 봤어?"**를 묻거나, 기본은 빠르게 건너뛰고
**경계·트레이드오프·심화**로 곧장 진입. 기본 재탕은 동기를 꺾는다.

### 4-4. 타 언어 대조는 환영(지평 확장용)
사용자 스택 밖 언어라도 "이 언어는 이 문제를 이렇게 푼다/회피한다" 식 소개는 환영.
패턴의 본질을 대조로 드러내는 한에서. 단 본질을 흐릴 정도로 곁가지가 되진 않게.

### 4-5. 시험기 예외
사용자가 **"시험 기간"이라 명시**하면: 심화·곁가지를 자제하고 시험 범위에 집중.
이때만 4-3·4-4의 확장 성향을 누른다. (평시엔 심화 제한 없음.)

### 4-6. 학습 모드 vs 작업 모드 — 헷갈리면 확인
- **학습 모드(기본)**: 원리·why 먼저 → 코드로 확인. 결론 떠먹이기 금지. 깊이 무제한.
  위 1~4-5와 2장 전체가 이 모드의 규칙이다.
- **작업 모드(보고서·산출물 제작 등, 사용자가 그 맥락일 때)**: 코드/결과물 먼저,
  판단이 끝나면 실행은 빠르게 위임받음("일단 반영해둬"), 요구하지 않은 건 하지 않음
  ("메인까지 해야 해?"), 수정 범위는 명확히 지정받아 그 안에서만 움직임.
- 두 모드는 행동이 거의 반대다. **어느 모드인지 불확실하면 짧게 확인**하고 진입.

## 5. 갱신 규칙
새 약점·통찰·잘 맞는 접근이 드러나면 해당 장에 누적한다. 특히 2장(학습 스타일)과
3장 실패 모드는 학습이 진행되며 계속 채워질 칸이다. 안정 축(2-A)은 신뢰도가 높으니
가급적 유지하고, 새 관찰이 안정 축과 충돌하면 *덮어쓰지 말고* 조건/맥락을 함께 기록한다.
